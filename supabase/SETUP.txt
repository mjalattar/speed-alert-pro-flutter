Speed Alert Pro — Supabase + Edge Function
==========================================

This folder holds **migrations/**, **functions/speed-limit-remote/**, and **config.toml** for the hosted Supabase project.

Run Supabase CLI commands from **this directory** (`speed-alert-pro-flutter/supabase`) — deploy once per hosted project.

0) Flutter app — keys and OAuth package name
   - Keys are passed with `--dart-define` (see project **README.md**), not `local.properties`.
   - Defaults live in `lib/config/app_config.dart` (`SUPABASE_URL`, `SUPABASE_ANON_KEY`, etc.).
   - For Google Sign-In on the **Flutter Android** build, add an OAuth **Android** client for package
     **com.example.speed_alert_pro** (see `android/app/build.gradle.kts` `applicationId`) and your
     debug/release SHA-1. Use the package name that matches the APK you install (see `applicationId`).

1) Create a Supabase project (https://supabase.com).

2) SQL: open SQL Editor, paste and run (in order):
    supabase/migrations/20250329120000_profiles_speed_cache.sql
    supabase/migrations/20260409000000_cache_key_coarse_grid_heading_bucket.sql
    supabase/migrations/20260410000000_profiles_subscription_fields.sql

   If the trigger fails on your Postgres version, replace the last line with:
   ... FOR EACH ROW EXECUTE PROCEDURE public.handle_new_user();

3) Auth — Google (required when SUPABASE_URL is set in the app):
   - Google Cloud: create an OAuth “Web application” client; note Client ID and Client secret.
   - Add an OAuth “Android” client for your app package (see §0) and your debug/release SHA-1
     (keytool -list -v -keystore ~/.android/debug.keystore -alias androiddebugkey -storepass android -keypass android).
   - Supabase → Authentication → Providers → Google: enable, paste Web client ID and secret.
     In “Client IDs”, include the Web client ID and Android client ID (comma-separated if the UI allows multiple).
   - You can turn off “Anonymous sign-ins”; the app signs in with Google only (ID token).

4) Edge Functions:
     a) `auth-check` — Authentication + RevenueCat subscription verification (independent from speed-limit-remote):
        Verifies JWT, checks trial period, calls RevenueCat, and stores result in `profiles.subscription_active` / `subscription_checked_at`.
        The app calls this on startup and after sign-in. No HERE key needed.
        Deploy:
          supabase functions deploy auth-check --no-verify-jwt
        Required secrets:
          supabase secrets set REVENUECAT_SECRET_API_KEY=your_rc_secret_key
          supabase secrets set RC_ENTITLEMENT_ID=premium

     b) `speed-limit-remote` — HERE proxy + cache (verifies JWT, checks subscription via profiles table):
        Deploy:
          supabase functions deploy speed-limit-remote --no-verify-jwt
        Required secrets:
          supabase secrets set HERE_API_KEY=your_here_rest_key
          supabase secrets set CACHE_TTL_HOURS=24
        (Alert cache key uses ~111 m–grid origin + 30° heading bucket + 3×3 neighbor probe;
         longer TTL cuts duplicate HERE transactions for many subscribers on the same road.)

     Shared secrets (both functions need these, set once):
       supabase secrets set SUPABASE_URL=https://YOUR_REF.supabase.co
       supabase secrets set SUPABASE_ANON_KEY=your_anon_key
       supabase secrets set SUPABASE_SERVICE_ROLE_KEY=your_service_role_key

    b) `auth-check` — Authentication + RevenueCat subscription verification (independent from speed-limit-remote):
       This function verifies the JWT, checks the trial period, calls RevenueCat,
       and stores the result in `profiles.subscription_active` / `subscription_checked_at`.
       The app calls this on startup and after sign-in.
       Deploy:
         supabase functions deploy auth-check --no-verify-jwt
       Required secret:
         supabase secrets set REVENUECAT_SECRET_API_KEY=your_rc_secret_key
         supabase secrets set RC_ENTITLEMENT_ID=premium
(Alert cache key uses ~111 m–grid origin + 15° heading bucket; longer TTL cuts duplicate HERE transactions
      for many subscribers on the same road. Keep compare flows in mind if you shorten TTL.)
   Deploy:
     supabase functions deploy speed-limit-remote --no-verify-jwt
   (JWT is verified inside the function via the user’s Bearer token.)

5) Optional — native Android `local.properties` (do not commit secrets to git):
   supabase.url=https://YOUR_REF.supabase.co
   supabase.anon.key=YOUR_KEY   (JWT anon or sb_publishable_… — if auth fails, use the legacy anon key from the dashboard)
   google.web.client.id=YOUR_WEB_CLIENT_ID.apps.googleusercontent.com
   revenuecat.public.api.key=goog_… or test_… (RevenueCat Android / test key)

   google.web.client.id is the OAuth 2.0 Web application client ID (used by Sign in with Google on device — not the Android client id).

   Settings → “Speed limit data source” lets you switch between on-device APIs and Supabase Edge without rebuilding.

   Optional: override HERE map/discover key (still used for WebView map + Discover in the app):
   here.api.key=YOUR_HERE_KEY

   When supabase.url is non-empty, the app sets USE_REMOTE_HERE=true: HERE routing for alerts
   and simulation goes through the Edge Function; a 3-day trial + RevenueCat entitlement `premium`
   are enforced on the server.

6) RevenueCat:
   - Create project, add Google Play app, entitlement id: premium (or set RC_ENTITLEMENT_ID on Edge).
   - Link subscription products (monthly/annual) to that entitlement.
   - The app identifies subscribers with the Supabase user UUID (Google account; logIn after sign-in).

7) Existing auth users without a profile row: run once in SQL:
   insert into public.profiles (id, trial_ends_at)
   select id, (now() at time zone 'utc') + interval '3 days'
   from auth.users
   on conflict (id) do nothing;
